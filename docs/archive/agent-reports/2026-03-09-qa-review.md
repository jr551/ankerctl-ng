## Findings

### KRITISCH (must fix before merge to release)

**K-001** `internal/web/handler/general.go:161-162` -- /video endpoint returns 501

Problem: The `/video` GET endpoint returns `http.StatusNotImplemented`. Python streams H.264 frames as chunked `video/mp4`. This is the only remaining 501 in the codebase.

Risk: Timelapse snapshot capture uses `videoLoopbackURL()` which calls `/video?for_timelapse=1`. Without this endpoint, all timelapse captures fail silently (ffmpeg gets no input). Additionally, direct video viewing from the browser is broken.

Fix: Implement chunked HTTP response generator that taps into VideoQueue's frame stream (similar to Python's `generate()` in `web/__init__.py:513-528`). Must handle: video_enabled check, VideoQueue auto-start, and graceful EOF on disconnect.

**K-002** `internal/pppp/protocol/channel.go:316-324` -- Write blocking loop ignores context

Problem: `Channel.Write()` with `block=true` loops on `c.Wait(250ms)` until all packets are ACKed. There is no way to cancel this wait via context. If the PPPP connection drops during a file upload, this loop blocks indefinitely (or until the entire service is forcibly killed).

Risk: Goroutine leak and hung uploads. The `PPPPService.Upload()` passes `req.ctx` but the underlying channel write ignores it entirely.

Fix: Add a `context.Context` parameter to `Channel.Write()` or provide a `WriteContext()` variant that checks `ctx.Err()` inside the wait loop.

### HOCH (should fix before release)

**H-001** `internal/web/middleware/auth.go:107-111` -- secureEquals leaks API key length

Problem: `secureEquals()` returns `false` immediately when `len(a) != len(b)`. This is a timing side-channel that leaks the API key length. While `subtle.ConstantTimeCompare` handles the byte comparison, the length check is not constant-time.

Risk: An attacker can determine the exact length of the configured API key through repeated requests, narrowing brute-force search space. Low practical risk for 16+ character keys but violates CLAUDE.md security mandate.

Fix: Remove the `len(a) != len(b)` early return. `subtle.ConstantTimeCompare` already handles different-length inputs correctly (returns 0 when lengths differ, in constant time).

**H-002** `internal/web/ws/pppp.go` (PPPPState handler) -- No probe/retry logic

Problem: The `/ws/pppp-state` WebSocket handler uses a simple ticker poll without the probe/retry/backoff logic from Python (`web/__init__.py:187`). Python has: dedicated probe thread, retry backoff (15s/60s after MAX_RETRIES=2), MQTT staleness check (30s), `probe_fail_count`, and immediate probe on WS connect.

Risk: Frontend shows stale PPPP connection state; user cannot tell if printer is actually reachable.

Fix: Port Python's probe logic with exponential backoff and MQTT staleness integration.

**H-003** `internal/service/mqttqueue.go:182-183` -- WorkerStart uses context.Background()

Problem: `WorkerStart()` calls `q.clientFactory(context.Background())` instead of using the service's run context. If the MQTT broker is unreachable, the 15-second HTTP client timeout blocks the entire service start with no way to cancel.

Risk: Service startup hangs for 15 seconds per retry attempt, regardless of whether shutdown was requested.

Fix: Accept a context parameter in `WorkerStart()` or derive from the worker's internal context.

**H-004** `internal/service/videoqueue.go:325-338` -- Type assertion chain in WorkerStart

Problem: `WorkerStart()` uses inline type assertions (`ppppLifecycle`, `videoHandlerRegistrar`) to discover PPPP service capabilities. This couples VideoQueue to PPPPService's concrete interface without any compile-time guarantee.

Risk: If PPPPService methods are renamed or removed, VideoQueue silently degrades with no error. Also, `time.Sleep(100ms)` polling loop in WorkerStart is not cancellable.

Fix: Define explicit interface types at package level. Pass the PPPP controller as a proper typed dependency, not a generic `VideoStreamController`.

**H-005** `internal/web/handler/bedlevel.go` -- Stub endpoints

Problem: `/api/printer/bed-leveling` and `/api/printer/bed-leveling/last` are stubs. The live endpoint sends a GCode M420 V but cannot parse the response. The "last" endpoint returns whatever grid data `MqttQueue` captured from ct=1073.

Risk: Feature gap. Users relying on bed leveling visualization get no useful data from the live endpoint.

Fix: Requires MQTT response parser for M420 grid format. Tracked as OI-013-BEDGRID-LIVE.

### MITTEL (fix in follow-up)

**M-001** `internal/service/pppp.go:500` -- discoverAndPersistPrinterIPs logs P2PDUID directly

Problem: In `config.go:500`, `slog.Warn("background IP discovery failed", "duid", p.P2PDUID, ...)` logs the DUID in plaintext. While DUID is not listed in CLAUDE.md's sensitive keys (auth_token, mqtt_key, api_key), the `logging.Redact()` function does redact `p2p_duid`. Inconsistency.

Risk: Minor information exposure. DUID is semi-public (broadcast on LAN) but should follow the project's redaction convention consistently.

Fix: Apply `logging.Redact()` consistently for DUID fields, or remove `p2p_duid` from the sensitive keys list if it is intentionally public.

**M-002** `internal/service/mqttqueue.go:559` -- time.Sleep in SendGCode blocks goroutine

Problem: `SendGCode()` calls `time.Sleep(100ms)` between GCode lines. This blocks the calling goroutine (typically an HTTP handler goroutine) and is not cancellable via context.

Risk: If a user sends 100 lines of GCode, the HTTP handler blocks for 10 seconds. During this time, the goroutine cannot be cancelled by request timeout.

Fix: Replace `time.Sleep` with `select { case <-ctx.Done(): return ctx.Err(); case <-time.After(100ms): }`.

**M-003** `internal/pppp/protocol/channel.go:64-73` -- Peek uses sync.Cond with timer-based wakeup

Problem: `Wire.Peek()` and `Wire.Read()` use `time.AfterFunc` + `sync.Cond.Wait()` for timed waits. This is correct but the timer goroutine acquires the same mutex as the main thread, creating contention under high throughput. Additionally, there is no context-based cancellation.

Risk: Under sustained video frame rate, the timer goroutine contention could add latency.

Fix: Consider using channels instead of sync.Cond for timed reads, which would also enable context cancellation.

**M-004** `internal/service/manager.go:157-185` -- keepVideoQueueRunning uses reflection

Problem: `keepVideoQueueRunning()` falls back to `reflect` to check a `VideoEnabled` field if the interface assertion fails. This is brittle and bypasses type safety.

Risk: The type assertion path (`videoEnabledGetter`) works correctly. The reflect fallback is dead code in practice but adds maintenance burden and confusion.

Fix: Remove the reflect fallback. The interface assertion is sufficient since all real VideoQueue implementations satisfy it.

**M-005** `internal/crypto/ecdh.go:24-30` -- mustDecodeHex panics

Problem: `mustDecodeHex()` uses `panic()` for a hardcoded hex constant. While this only runs during package init and the input is a compile-time constant, CLAUDE.md mandates "No panics on production paths".

Risk: Theoretical. The panic can never trigger unless the source code is modified to contain invalid hex.

Fix: Acceptable as-is since it is an init-time constant. Add a comment clarifying this is intentional for package-level constant initialization.

**M-006** `internal/web/handler/debug.go:54,73` -- Ignored JSON decode errors

Problem: `DebugConfig` and `DebugSimulate` both use `_ = json.NewDecoder(r.Body).Decode(&payload)` without checking the error. An invalid JSON body silently produces zero values.

Risk: Dev-mode only, so impact is limited. But it means malformed requests succeed silently, complicating debugging.

Fix: Return 400 on decode error for consistency with other endpoints.

### NIEDRIG (nice-to-have)

**N-001** `internal/web/handler/general.go:236` -- ConfigLogin logs email in plaintext

Problem: `slog.Info("cloud login successful", "email", email, ...)` logs the user's email address in plaintext. While email is not in the `sensitiveKeys` list, it is PII.

Risk: Email appears in server logs. Minor privacy concern.

Fix: Redact or omit email from log output.

**N-002** `cmd/find_anker/main.go` -- No test file

Problem: The `find_anker` command tool has no tests.

Risk: Tool is standalone and simple. Low risk.

**N-003** `internal/pppp/protocol/packet.go:46-47` -- Duplicate type constants

Problem: `TypePunchPktEx = 0x41` and `TypeP2pRdyEx = 0x42` are aliases for `TypePunchPkt` and `TypeP2pRdy` respectively. These are never used in the codebase but could cause confusion.

Risk: None functional. Documentation debt.

Fix: Add comments explaining the aliases exist for Python parity, or remove them.

## Positive Observations

1. **Excellent layering discipline.** Zero import violations across all 22 packages. The dependency graph strictly follows CLAUDE.md's rules. Interface-based decoupling is clean and minimal.

2. **Security posture is strong.** `logging.Redact()` is applied consistently. No secrets appear in any `slog.*` call. XSS prevention via `html.EscapeString()` in filament handlers. Path traversal checks in timelapse and debug log endpoints. Config directory created with 0700. API key comparison uses `subtle.ConstantTimeCompare`.

3. **Protocol implementation is bit-exact.** The PPPP curse/decurse shuffle table, CRC16, AES-256-CBC with fixed IV, MQTT packet framing (63-byte M5 header / 24-byte M5C header), and CyclicU16 wraparound all match the Python reference. Table-driven tests validate round-trip correctness.

4. **Service lifecycle is well-designed.** BaseWorker with BindHooks pattern provides consistent init/start/run/stop semantics. Reference-counted Borrow/Return in ServiceManager auto-starts and auto-stops services. The holdoff/restart mechanism prevents rapid restart loops.

5. **All previously identified critical bugs (#1-#8) from the March 6 bugfix plan are fixed.** The ws/ctrl protocol now accepts Python-format payloads, light control routes through PPPP, P2PCommand uses correct JSON shape, GCode sends line-by-line with cmdLen, and file transfer respects StartPrint flag.

6. **Cloud login flow is fully implemented.** ECDH password encryption, CAPTCHA handling, FDM list + DSK key fetching, IP preservation across logins, and background LAN discovery are all present and match Python behavior.

7. **SQL queries are parameterized.** The `fmt.Sprintf` in `UpdateFilament` builds column names from a hardcoded whitelist (`filamentWritableFields`), not user input. All values use `?` placeholders.
