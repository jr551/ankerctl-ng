## TASK 1: Race Condition Fix — `/ws/ctrl`

### Root Cause (Two Separate Races)

**Race 1 — Concurrent writers on conn:**
`Ctrl()` spawned two goroutines that both wrote to the same `*websocket.Conn`:
- The auth goroutine called `conn.WriteControl()` (via `closePolicyViolation`) on auth failure.
- The write pump (`writePump`) called `conn.WriteJSON()` and `conn.WriteControl()` (pings) concurrently.

gorilla/websocket explicitly prohibits concurrent writers. Race detector confirmed this at `conn.WriteControl` vs `conn.WriteJSON`.

**Race 2 — Concurrent readers on conn:**
`writePump` unconditionally spawned `readPump` via `go h.readPump(conn, readDone)`. But `Ctrl` also reads from `conn` in the auth goroutine (`authenticateCtrl`) and `ctrlReadLoop`. This produced two goroutines calling `conn.ReadMessage()` simultaneously. Race detector confirmed this at `bufio.(*Reader).Peek` vs `bufio.(*Reader).Read`.

### Fix

Replaced the generic `writePump` call in `Ctrl` with a dedicated `ctrlWritePump` that:
1. Never spawns a separate `readPump` — `Ctrl` manages its own single reader goroutine.
2. Is the **sole writer** of `conn` for the entire lifetime of the handler.

`authenticateCtrl` no longer calls `conn.WriteControl()` directly. On auth failure it enqueues a `ctrlCloseFrame` sentinel value (a new unexported struct type) into the buffered `out` channel. `ctrlWritePump` detects this type and issues the policy-violation close frame.

**Priority drain pattern** in `ctrlWritePump` ensures the close frame is always delivered even when `ctx.Done()` and `readDone` fire simultaneously after the auth goroutine enqueues the frame and calls `cancel()`. The pump checks `out` first in a non-blocking select at the top of every iteration, and drains one final message from `out` in the `ctx.Done()` and `readDone` cases.

### Goroutine Topology (Post-Fix)

```
Goroutine A (reader): authenticateCtrl → ctrlReadLoop
    - sole reader of conn
    - communicates to B via out channel (chan any, buffered 32)
    - signals shutdown via readDone (chan struct{})

Goroutine B (writer): ctrlWritePump
    - sole writer of conn
    - handles ctrlCloseFrame sentinels, JSON messages, ping keepalives
    - exits on ctx.Done(), readDone, or write error
```

### Result

`go test -race -count=5 ./internal/web/ws/...` passes clean.
`go test -race ./...` passes clean.

## TASK 2: Python Compliance Check

### `/ws/ctrl` — Initial State Messages

**Finding:** Python sends two messages on connect (after auth in our case):
1. `{"ankerctl": 1}` — handshake, tells client the connection is ready.
2. `{"video_profile": <current_profile>}` — current video profile, default `"sd"`.

Go only sent `{"auth": "ok"}`. The initial state messages were missing.

**Fix:** Added `sendCtrlInitialState()` called immediately after auth succeeds and `{"auth": "ok"}` is enqueued. It sends `{"ankerctl": 1}` then `{"video_profile": <profile>}`. If the `videoqueue` service implements `CurrentProfile() string`, the actual current profile is used; otherwise `"sd"` (the Python default `VIDEO_PROFILE_DEFAULT_ID`) is sent.

Message order on successful auth: `{"auth":"ok"}` → `{"ankerctl":1}` → `{"video_profile":"sd"}`.

**Test updated:** `TestCtrlHandler_AuthAndCommands` now drains the 2 initial state messages before reading command responses.

### `/ws/pppp-state` — `dormant` State

**Finding:** `dormant` IS a valid state in Python (`web/__init__.py` lines 171, 265-266). It means the PPPP service is not running (video/timelapse not active). The Go handler already emits `"dormant"` correctly. No change needed.

### Apprise API Path

**Finding:** Python `_notify_url()` builds `{server_url}/notify/{key}` (apprise_client.py lines 204-213). Go `notifyURL()` does the same. Correct, no change needed.

### Event Names

**Finding:** Python `events.py` defines: `print_started`, `print_finished`, `print_failed`, `gcode_uploaded`, `print_progress`. Go has all five plus extensions (`print_paused`, `print_resumed`). The base five match exactly. No change needed.

### Notification Type Values

**Finding:** Python's `AppriseClient._post()` sends only `title` and `body` — no `type` field. Go adds a `type` field (`info`/`success`/`failure`). This is a Go extension, not a regression. Apprise accepts `failure` as a valid type value per its API spec. No change needed.
